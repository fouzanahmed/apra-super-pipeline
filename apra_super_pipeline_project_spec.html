Full project specification for the Australian superannuation analytics pipeline

Australian Superannuation Analytics Platform
End-to-end data pipeline ingesting APRA's public regulatory data ($4.5T AUM industry) — automated ingestion, dbt transformations, AI-powered querying, and a live Power BI dashboard. Deployed on AWS with Docker and CI/CD.
Python dbt Airflow AWS RDS + EC2 + S3 Docker Power BI FastAPI GitHub Actions CI/CD OpenAI / Claude API PostgreSQL PySpark
1
Data source — what you're ingesting
Free, public, no API key
APRA Quarterly MySuper Stats
Per-fund fees, returns, asset allocation by quarter. 8MB Excel, back to 2013. This is your core fact table.
apra.gov.au/quarterly-superannuation-statistics
APRA Quarterly Fund-Level Stats
Member demographics + investments by asset type per fund. Released June 2024 — very few portfolios use this yet.
apra.gov.au/quarterly-fund-level-statistics
APRA Annual Fund Bulletin
Long historical view: fees, expenses, membership trends per fund since 2015. Use for trend analysis.
apra.gov.au/annual-superannuation-bulletin
2
Ingestion — Python + AWS S3
You learn: S3, boto3, pandas
  • 2.1
    Download script — Python script using requests to pull APRA Excel files. Clean column names (snake_case), fix multi-header rows, validate row counts. ~60 lines.
  • 2.2
    Upload to S3 — Push raw Excel files to an S3 bucket (s3://super-pipeline/raw/) using boto3. This is your data lake layer. Keeps a dated archive each run.
  • 2.3
    Load to Postgres — Read from S3, parse with pandas, load to raw.apra_mysuper table on AWS RDS (Postgres). Use upsert logic — idempotent, safe to rerun.
3
Transformation — dbt on Postgres
You learn: dbt, SQL modeling, data testing
staging/stg_apra_mysuper.sql
Clean raw table: cast types, rename columns, filter nulls. One model, ~20 lines. First dbt model you write.
mart/fund_performance.sql
1yr/3yr/5yr returns per fund. RANK() among peers. Fee percentile. Window functions — this is interview SQL.
mart/fee_vs_return.sql
"Value for money" table: high fees + low returns flagged. What financial journalists publish annually.
mart/member_flow.sql
Net member inflows/outflows per fund per quarter. Which funds are losing members and why.
Add dbt tests on every model: not_null, unique, accepted_values, relationships. This becomes your CI/CD test suite.
4
Orchestration — Airflow on EC2
You learn: DAGs, scheduling, Docker Compose
  • 4.1
    Docker Compose setup — One docker-compose.yml file spins up Airflow + Postgres locally. Anyone can clone and run. Study Eric's repo for this pattern exactly.
  • 4.2
    Weekly DAG — Every Monday 8am: download APRA files → upload to S3 → load to RDS → run dbt run → run dbt test → send Slack alert on failure.
  • 4.3
    Deployed to AWS EC2 — t2.micro free tier. Airflow runs there permanently. This is the "production" environment — not just localhost.
5
AI layer — NL-to-SQL query API
You learn: FastAPI, LLMs, prompt engineering
  • 5.1
    FastAPI backend — One endpoint: POST /query accepts a plain English question. You already have FastAPI on your CV — this extends it with AI.
  • 5.2
    Schema-aware prompt — Send the question + your dbt mart table schemas to Claude/GPT-4. Prompt: "Given these tables, write a SQL query to answer: {question}". Return the SQL + result as JSON.
  • 5.3
    Example queries it handles — "Which fund had the best 5yr return under 0.5% fees?" / "Which 3 funds lost the most members last quarter?" / "Compare AustralianSuper vs Hostplus fees since 2020."
This is the AI component. It's genuine text-to-SQL, not a notebook — and it's built on real data, which makes it demonstrable in interviews.
6
Visualisation — Power BI connected to RDS
You learn: DirectQuery, DAX basics
Page 1: Fund rankings
Bar chart: top 20 funds by 5yr return. Slicer by fund type (industry/retail/public sector). This is page 1.
Page 2: Fee vs return scatter
X = admin fees %, Y = 1yr return. Quadrant labels: "best value", "worst value". Every fund is a dot.
Page 3: Member flow trends
Line chart: net member inflows per fund over 8 quarters. Which funds are growing vs bleeding members.
Connection method
DirectQuery from Power BI Desktop to AWS RDS via ODBC. Always live data — not an Excel import.
Publish to Power BI Service (free for individuals) so you have a shareable link — not just a screenshot. This is the portfolio-ready output.
7
CI/CD — GitHub Actions
You learn: YAML workflows, automated testing
  • 7.1
    PR check workflow — On every pull request: run ruff Python linter + dbt compile to validate SQL. Fails the PR if anything breaks. File: .github/workflows/pr_check.yml.
  • 7.2
    dbt test workflow — On merge to main: run dbt test against a test database. Catches data quality regressions. File: .github/workflows/dbt_test.yml.
  • 7.3
    Docker build check — Confirm the Docker image builds cleanly on every push. Prevents broken deployments. ~10 lines of YAML.
Total CI/CD code: ~60 lines of YAML across 3 files. The GitHub Actions badge on your README (green checkmark) is visible to every recruiter who visits.
8
PySpark bonus layer (India market)
Add after core is done
Write a PySpark script (spark/quarterly_aggregation.py) that reads the raw APRA parquet files from S3 and produces the same fund-level aggregations as dbt — but using Spark DataFrames. Run it locally with pyspark or on an EMR Serverless job. The output is intentionally redundant; the point is to show Spark on your CV with real code. Add it to your Airflow DAG as an optional parallel branch.
